# Interview Questions & Scenarios

## Tier 1: Senior Engineer (L5)

### Q1: What is LLM-as-judge, and what are the top three biases you need to account for?

**Model Answer**: LLM-as-judge uses a powerful language model (Claude, GPT-4o) as an automated evaluator to score or compare LLM outputs on dimensions like faithfulness, relevance, or helpfulness — using a structured rubric and often requiring the judge to reason before scoring. The three most important biases: (1) **Verbosity bias** — longer responses are rated higher regardless of content quality, because the judge conflates detail with quality. Mitigation: explicitly state in the judge prompt "do not be influenced by response length." (2) **Position bias** in pairwise evaluations — the first response presented tends to win more often than chance. Mitigation: run each comparison twice with positions swapped and average the results. (3) **Self-preference** — models tend to rate responses from models with similar style or training higher; GPT-4 as judge slightly favors GPT-4 responses. Mitigation: use a different model family as the judge, or use an ensemble of judges and take majority vote.

### Q2: Explain the difference between faithfulness and answer correctness in RAG evaluation.

**Model Answer**: Faithfulness measures whether the generated answer is grounded in the retrieved context — every claim in the answer can be traced to the context. It's a reference-free metric (no ground truth needed) that catches hallucination: did the model make things up that aren't in the context? Answer correctness measures whether the answer is factually right compared to a ground truth answer — it's a reference-based metric requiring a labeled dataset. You can have high faithfulness with low correctness (model accurately reflects wrong documents), or low faithfulness with high correctness (model gives the right answer but from parametric memory, not the context). In production RAG, faithfulness is more operationally important because you can't always have ground truth, and faithfulness directly measures the hallucination risk. Answer correctness is critical for evaluating the full pipeline quality during offline evaluation with a labeled test set.

### Q3: How would you build a minimal eval pipeline for a new LLM feature before it ships to production?

**Model Answer**: Three components. First, a golden dataset of 100–200 representative queries with expected behavior — sampled from similar existing queries, edge cases identified during development, and adversarial cases. Second, an automated eval script that runs the feature against the golden set and computes relevant metrics: LLM-as-judge for quality dimensions (faithfulness, relevance) and rule-based checks for schema/safety constraints. Third, a baseline — run the eval against the current production system and store the scores. The new feature must match or exceed baseline on all metrics within a 3% tolerance. Wire this into CI/CD so it runs on every PR that changes prompts, retrieval, or model configuration. The whole thing can be operational in a day; the bigger investment is building a representative golden dataset, which typically takes a few days of careful curation.

### Q4: What is context_precision vs. context_recall in RAGAS, and when does each fail?

**Model Answer**: Context precision measures what fraction of retrieved chunks were actually relevant to answering the question — it catches retrieval noise (you retrieved 10 chunks but only 2 were useful). Context recall measures what fraction of the information needed to answer the question was present in the retrieved chunks — it catches retrieval misses (the relevant document wasn't retrieved at all). Context precision fails when the embedding model or retriever is poorly calibrated to the domain, returning tangentially related documents. Context recall fails when the golden dataset contains questions requiring rare, specific information that the retriever doesn't surface — common when the corpus has changed since indexing, or when queries use terminology different from the document vocabulary (a case where hybrid BM25+dense search outperforms pure embedding retrieval).

### Q5: What is behavioral testing for LLMs, and how does it differ from standard metric evaluation?

**Model Answer**: Behavioral testing (from Ribeiro et al.'s CheckList) tests specific model behaviors with targeted test cases rather than average performance on a dataset. It's structurally similar to software unit tests: each test verifies that the model behaves correctly under a specific condition. Three test types: Minimum Functionality Tests verify basic capabilities (the model correctly identifies capitals); Invariance Tests verify that outputs don't change when inputs are perturbed irrelevantly (adding "please" or changing capitalization shouldn't change factual answers); Directional Tests verify that specific input changes produce expected output changes (adding more context should not hurt answer quality). Standard metric evaluation gives you an average quality score — useful for tracking trends. Behavioral tests give you coverage of specific failure modes — useful for CI/CD gates. Both are complementary: metrics tell you how good you are on average; behavioral tests tell you whether you've broken specific important behaviors.

### Q6: A RAG system has faithfulness 0.91 in offline evaluation but users are complaining about wrong answers. What do you investigate?

**Model Answer**: The gap between offline and online quality is a dataset distribution problem. I'd investigate in this order: First, check if the offline golden set was sampled from the same user query distribution as current production traffic — if it was sampled at launch and usage patterns have shifted (new topics, new user types), the golden set no longer represents reality. Second, sample 50 recent low-satisfaction production queries (from thumbs-down, escalation, or abandonment signals), run faithfulness eval on those specifically, and compare to the golden set score — if the production sample scores 0.65, the dataset is stale. Third, look at the failure cases themselves: are they a specific query type (multi-hop, numerical, temporal)? A specific document type that's newly ingested? A specific user segment? Fourth, add the production failure cases to the golden set and re-run — this both measures the true extent of the problem and prevents future regressions on this query type.

## Tier 2: Staff Engineer (L6)

### Q7: Design an eval pipeline for a multi-step AI agent that books travel itineraries using 5 tools (search, weather, calendar, pricing, booking).

**Model Answer**: Agent eval requires three levels of evaluation, not one. Task level: binary success rate — did the agent produce a valid, complete itinerary? This requires a test harness with mock implementations of all 5 tools that return controlled responses, and 50–100 test scenarios with known expected outcomes (destination, dates, constraints). Trajectory level: given the task succeeded, was the path efficient? Measure: number of tool calls, number of unnecessary redundant calls, correct tool selection for each step (e.g., checking weather before booking activities, not after). Use an LLM judge that sees the full trajectory and rates reasoning quality at each step. Robustness level: inject controlled failures — tool returns error, pricing API times out, weather data unavailable — and measure whether the agent recovers gracefully or fails catastrophically. Each level catches different failure modes. Task success alone misses agents that get to the right answer via an inefficient or fragile path. Report all three; gate CI on task success rate ≥ 0.85 and tool selection accuracy ≥ 0.90.

### Q8: Your LLM judge has 0.77 pearson correlation with human labels. The product team wants to ship. How do you decide whether this is acceptable?

**Model Answer**: 0.77 is borderline — substantial agreement but below the 0.80 threshold I'd want for a production quality gate. The right answer depends on three things. First, what is the judge being used for? If it's a CI gate blocking deployments, a miscalibrated judge causes either false positives (blocking good changes) or false negatives (shipping regressions) — both are costly. For this use, I'd insist on recalibration before shipping. If it's a monitoring signal for trend detection, 0.77 is acceptable — it will catch large regressions even if not precise. Second, what's the source of disagreement? If the judge and humans disagree on borderline cases (score 3 vs 4) but agree on clear passes and failures, the practical impact is small. If they disagree on obvious failures (judge gives 4, human gives 1), the judge is unreliable. Analyze the disagreement cases specifically. Third, can we operate with a conservative threshold? If I set the alert threshold at 0.65 instead of 0.75, a judge that slightly over-scores by 0.15 would still trigger alerts on real failures. My recommendation: ship with conservative thresholds and a plan to recalibrate within 30 days on a larger human annotation sample.

### Q9: How do you detect and handle benchmark contamination when evaluating a fine-tuned model?

**Model Answer**: Benchmark contamination occurs when the benchmark's test examples appear in the model's training data, inflating scores. Four detection approaches. First, n-gram overlap: compute the fraction of test questions where significant n-gram overlaps (10+ grams) appear in the training corpus — if >5% overlap, contamination is likely. Second, canary injection: embed unique, novel questions with known answers into your benchmark before sharing it; if the model answers them correctly at an anomalously high rate, the benchmark leaked into training. Third, behavioral gap analysis: contaminated models often score high on exact benchmark questions but fail similar, rephrased variants. Test the model on paraphrases of benchmark questions — a 30%+ accuracy drop on paraphrases signals contamination rather than genuine capability. Fourth, use dynamic benchmarks like LiveCodeBench (monthly new problems) or internal hold-out sets with a creation date after the model's training cutoff. For fine-tuning specifically: always maintain a held-out evaluation set that is never included in fine-tuning data, stored in a separate access-controlled location, and versioned with a clear policy prohibiting its use in training.

### Q10: Design the eval infrastructure for a team of 20 engineers shipping LLM features weekly. What processes and tooling do you put in place?

**Model Answer**: Four-layer infrastructure. Tooling: Braintrust or LangSmith for eval result tracking, dataset management, and experiment comparison — not spreadsheets. RAGAS + DeepEval for eval logic in Python. GitHub Actions for CI integration. A shared internal `eval_utils` library so every team uses the same judge prompts, metrics, and baselines — avoiding eval fragmentation. Process: every feature team owns an eval dataset for their feature area (minimum 200 samples, reviewed quarterly). Shared golden set of 500 cross-cutting queries maintained by a platform team. Every PR that touches prompts/retrieval/models runs the eval gate (200 samples, 5 minutes, ~$0.10). Weekly comprehensive eval run (1000 samples) triggered on schedule. Baselines: stored in Git alongside the golden dataset; baseline updates require team lead approval and a comment documenting why the change is expected. Roles: one "eval owner" per squad responsible for dataset quality and baseline governance — this is a rotating responsibility, not a separate team. Onboarding: new engineers write at least 20 eval cases in their first sprint, so they understand the eval system before shipping features.

### Q11: Explain the Eval-Driven Development workflow — how does it change the typical LLM development loop?

**Model Answer**: Eval-Driven Development (EDD) inverts the typical "build then test" loop into "define eval → build → measure." The workflow: (1) Before writing any prompt or retrieval code, define what success looks like in measurable terms — write 20–30 test cases with expected outputs and the rubric for judging them. (2) Run the eval on the current system (baseline). (3) Iterate on prompts, retrieval, or model — each iteration is measured against the eval, not eyeballed on a few examples. (4) When the eval score reaches target, ship. The compounding benefit: the eval set grows with every iteration (add failure cases you discover), creating an ever-richer regression suite. This changes the developer experience in three ways: prompt changes are verified by data, not intuition; A/B experiments between prompting strategies are settled by scores, not debate; and new engineers can contribute safely because the eval gate catches regressions. The anti-pattern it prevents: "I changed the prompt and it looks better on my 5 examples" shipping to production and breaking 15% of queries the developer didn't test. DSPy takes this further — it can automatically optimize prompts against an eval set, making the eval the optimization target directly.

### Q12: What is the key challenge in evaluating long-context LLM tasks (100K+ token inputs), and how do you address it?

**Model Answer**: Three interrelated challenges. First, **ground truth is expensive**: for a 100K-token document, writing a representative evaluation set requires annotators to read and understand massive documents — expert annotator time is expensive and slow. Mitigation: synthetic data generation (have the LLM generate question-answer pairs from the document, then have humans validate them rather than write from scratch — 5x faster). Second, **"lost in the middle" detection**: models degrade on evidence in the middle of long contexts, but standard metrics don't tell you where in the context the model failed. Use needle-in-a-haystack evals: embed a specific fact at a known position (10%, 50%, 90% through the document) and verify the model can recall it. HELMET benchmark (2025) is specifically designed for this. Third, **metric reliability degrades**: RAGAS-style chunked context evals assume retrievable chunks; full-document evals have the entire document as context, so context_precision and context_recall don't apply. Instead, use reference-based correctness (was the specific fact from the document included in the answer?) and faithfulness (is the answer grounded in the document?), both computed with the full document as context — which requires a judge model with comparable or longer context than the model being evaluated.

## Tier 3: Principal Engineer (L7+)

### Q13: A VP asks why you're spending engineering time on eval infrastructure when the team could be shipping features. How do you make the ROI case?

**Model Answer**: The ROI argument has three parts. First, the cost of not having evals. Pull the last 6 months of production incidents involving the LLM system — classify them by whether the regression was detectable via eval. In my experience, 70–80% of LLM quality incidents are catchable with a faithfulness or relevance eval gate. Quantify: how many hours did on-call engineers spend? What was the user impact (escalation rate, churn, support tickets)? If a $1/run eval gate catches $50K/quarter in incident response costs, the ROI is obvious. Second, feature shipping velocity. Without evals, engineers are afraid to change prompts — a prompt change requires manual spot-checking that takes hours and is still unreliable. With an eval gate, a prompt change is validated in 5 minutes. Teams with eval infrastructure ship prompt changes 3–5x faster than teams without. Third, the eval infrastructure itself is a feature. Customers (enterprise, regulated industries) increasingly ask "how do you measure and guarantee the quality of your AI system?" A robust eval pipeline is a sales and compliance asset. The meta-point: eval infrastructure is not a tax on feature shipping — it's the foundation that makes fast, safe feature shipping possible. Without it, you're flying blind and slowing down with each incident.

### Q14: Design an evaluation framework for a customer-facing AI system that operates in a regulated industry (healthcare or financial services) where you must be able to audit every AI output.

**Model Answer**: Regulated contexts require five additions to standard eval infrastructure. First, **immutable audit trail**: every AI output must be logged with the full context — input, retrieved documents, prompt, model response, model version, and timestamp — in a tamper-evident store (AWS CloudTrail + S3 Object Lock with WORM, or similar). This is not eval data; it's compliance infrastructure. Second, **deterministic safety gates**: LLM-as-judge is non-deterministic and cannot be the final safety gate. Implement rule-based safety checks (PII detection, medical claim classifier, regulatory language detector) that run as hard gates — if these fail, the response is blocked regardless of quality score. Third, **human review sampling with SLA**: for high-stakes outputs (medication dosage information, financial advice thresholds), route a configurable percentage to human reviewers with a guaranteed review SLA. The eval system must track review queue depth and reviewer agreement with the AI. Fourth, **regulatory-aligned metrics**: define metrics that map to the specific regulatory requirement — for HIPAA, measure PII leakage rate; for FDA AI/ML guidance, measure clinical accuracy on validated test sets; for financial services, measure factual accuracy on regulatory disclosures. These metrics must be reported to compliance officers, not just the engineering team. Fifth, **model change governance**: any change to the prompt, model version, or retrieval system must go through a formal evaluation process with documented approval — not just CI gate pass. Define what constitutes a "material change" requiring re-validation vs. a non-material change. The eval system version, golden dataset version, and baseline are part of the change record.

### Q15: The industry is moving toward "evals as optimization targets" (DSPy, automated prompt optimization). What are the risks of this approach, and how do you mitigate them?

**Model Answer**: Automated prompt optimization directly against an eval set — as in DSPy's `Teleprompter`, or OPRO (Optimization by PROmpting) — is powerful but introduces Goodhart's Law at scale: when an eval becomes the optimization target, it ceases to be a good measure of what you care about. Three specific risks. First, **eval overfitting**: the optimizer finds prompts that score perfectly on the eval set but fail on held-out distribution. Mitigation: maintain a strict held-out test set (never used for optimization), evaluate final prompts on it before shipping, and use cross-validation within the optimization loop. Second, **adversarial exploitation of the judge**: if the judge model is known, the optimizer can find prompts that exploit judge biases (verbosity, position, sycophancy) rather than genuinely improving quality. Mitigation: use a judge ensemble (multiple models), keep the judge model a black box to the optimizer, and validate on a held-out human evaluation sample. Third, **distribution shift invisibility**: optimized prompts perform well on today's query distribution but may fail as the distribution shifts. The eval set must be periodically refreshed from production traffic, and re-optimization should not automatically use the new eval set without human review of whether the new optimum is genuinely better or just optimized to a different distribution. The meta-principle: automated optimization amplifies both the quality of your eval and its flaws. Invest in eval quality before applying automated optimization — a good optimizer applied to a bad eval produces an impressively bad system.

### Q16: You're evaluating a new reasoning model (o3-class, "thinking" models) where the model produces a long chain-of-thought before the final answer. How does eval design change?

**Model Answer**: Chain-of-thought models introduce a new evaluable artifact — the reasoning trace — and change the relationship between intermediate steps and final outputs. Three design changes. First, **separate trace quality from answer quality**: the final answer may be correct even when the reasoning chain contains errors (model "gets lucky"), or incorrect despite correct reasoning (model fails to execute its own plan). Evaluate both independently: trace correctness (are the reasoning steps valid? does each step follow from the previous?) and answer correctness (is the final answer right?). Trace quality is a strong leading indicator of model reliability — high answer quality with poor trace quality suggests the model is not truly reasoning but pattern-matching. Second, **cost-aware eval design**: thinking models generate 5–20x more tokens than non-thinking models. Running LLM-as-judge on the full trace is expensive. Use a tiered approach: rule-based checks on the final answer (fast, cheap), LLM-as-judge on a 10% sample of full traces (expensive, high signal), automated logical consistency checks on the trace structure (medium cost). Third, **new failure mode: trace-answer inconsistency**: the model's trace says "the answer is X" but the final answer is Y. This is a distinct, high-severity failure mode — the model overrides its own reasoning. Build a specific eval that detects trace-answer inconsistency and report it separately, as it indicates reasoning instability rather than knowledge gaps.
